10GLOSARIO

BPM: Business Process Management es una disciplina que combina calidad
de software y la habilidad del negocio para la mejora de procesos y facilitar su
innovación. Dichos procesos son susceptibles de modelar, automatizar,
integrar, monitorizar y mejorar en el tiempo. La gestión de procesos de negocio
integrada con la tecnología permite alcanzar los objetivos de negocio mediante
la optimización de los procesos, automatización de tareas, respuesta rápida

ante el cambio, todo monitoreándose con el fin de poder tomar decisiones [1]

BPMN: Business Process Modeling Notation fue construida como una notación
gráfica fácilmente entendible que ayuda a enfocarse en la ejecución de
procesos de negocio. Permite soporte fuerte para la especificación de controles
de flujo y está estructurada gráficamente [2]. Además, permite ser un puente

entre el diseño de procesos y su respectiva implementación [3].

BPMS: Business Process Management System, software para la
automatización de procesos de negocio ya modelados en una notación, por
ejemplo BPMN, transftormándolos en un lenguaje ejecutable como BPEL.
Permiten efectuar simulaciones, integraciones con otros sistemas e

implementar mejoras [4].

ELICITACIÓN DE REQUISITOS: es una de las etapas iniciales del proceso de
desarrollo de software. Abarca la comunicación entre el equipo de desarrollo y
los stakeholders, clientes y usuarios finales. La elicitación tiene como objetivo
capturar, recolectar y descubrir las necesidades de los usuarios que permitirán

al equipo de desarrollo, definir qué debe hacer el sistema a desarrollar [5].

11ITERACIÓN: es un conjunto de periodos de tiempo en un proyecto, dentro del

cual se produce un software o servicio, documentación y demás entregables

[6].

MANIFIESTO ÁGIL: documento que contiene 4 valores y 17 principios,
propuestos por 17 industriales del software, en donde se establece el

desarrollo ágil conservando la calidad del software [7].

STAKEHOLDERS: Cualquier grupo o individuo identificable que pueda afectar
el logro de los objetivos de una organización o que es afectado por el logro de
los objetivos de una organización (grupos de interés público, grupos de
protesta, agencias gubernamentales, asociaciones de comercio, competidores,

sindicatos, así como segmentos de clientes, accionistas y otros) [8].

PRIME FACES: es un conjunto de componentes JSF de código abierto con
varias extensiones, contiene un amplio conjunto de componentes (editor
HTML, de diálogo, de Autocompletar, gráficos y muchos más), capacidad de
Ajax basado en APIs estándar JSF Ajax, ligero, sin necesidad de configuración
y no hay dependencias requeridas, soporte de empuje a través de Marco
Ambiente, kit de interfaz de usuario móvil para crear aplicaciones web para

móviles y una amplia documentación [9].

JSF: es un framework MVC (Modelo-Vista-Controlador) basado en el API de
Servlets que proporciona un conjunto de componentes en forma de etiquetas
definidas en páginas XHTML mediante el framework Facelets. Facelets se
define en la especificación 2 de JSF como un elemento fundamental de JSF
que proporciona características de plantillas y de creación de componentes
compuestos. Antes de la especificación actual se utilizaba JSP para componer

las páginas JSF[10].

12FRAMEWORK: Un framework para aplicaciones es un software o conjunto de
librerías, que está diseñado para dar soporte al desarrollo y en general a la
construcción de cualquier aplicación. Entonces un framework trata de facilitar
aquellas actividades comunes realizadas durante el desarrollo de la aplicación,
como por ejemplo: acceso a la base de datos, uso de plantillas, manejo de
sesiones, separación de aspectos de programación; además de promover la re

utilización de códigol|11|].

13INTRODUCCIÓN

La gestión de las exigencias y los requisitos de los clientes requiere que las
organizaciones se soporten en herramientas de software, las cuales pueden ser
adquiridas o desarrolladas a medida. En ese sentido, el desarrollo de software se
constituye como parte importante del capital mundial, debido a que permite la
evolución de las empresas y la administración del conocimiento [12]. Basado en lo
anterior, la industria del software interviene de manera importante en la "nueva
economía", pues es una industria blanca, que no contamina y que genera fuentes

de trabajo bien remuneradas [13].

En este escenario, para desarrollar software competitivo se requiere estar
comunicado con el mundo, tener acceso a las últimas tecnologías y aplicar
metodologías adecuadas a cada proyecto [12] [14]. Teniendo en cuenta que las
condiciones y limitaciones de cada proyecto son diferentes, se hace necesario
adaptar las metodologías a cada situación. Las metodologías ágiles han sido
especialmente pensadas para proyectos en que el desarrollo de software tiene
plazos reducidos, requisitos volátiles, y/o basados en nuevas tecnologías [14] [15].
Una de estas metodologías es eXtreme Programming (XP), esta se centra en
potenciar las relaciones interpersonales como punto clave para el éxito en el
desarrollo de software, promoviendo el trabajo en equipo, el aprendizaje de los
desarrolladores y el buen clima de trabajo; basándose en retroalimentación

continua entre el cliente y el equipo de desarrollo [15].

Como muestra de lo anterior, Business Process Modeling (BPM) se ha convertido
en una herramienta que apoya la elicitación de requisitos, mezclándose en el ciclo
de vida del desarrollo de software, para generar un mejor entendimiento entre los
líderes de procesos y el personal técnico de desarrollo del proyecto. Además, se

ha demostrado que la inclusión de BPM en los procesos de desarrollo de software,

14disminuye los tiempos y los costos, permite el desarrollo de actividades paralelas
y la definición de roles dentro del proyecto, posibilitando la participación activa del

usuario final en el proceso [16].

Ante este panorama el propósito de este trabajo fue analizar la situación actual
de la empresa SOMEC (Sociedad Mecánica) S.A. en cuanto a la gestión de
los activos de información, con el propósito de servir de referencia y apoyo
en la definición de estrategias que permitieran dar una solución informática
que aplique una correcta gestión de los activos de información con que se
cuentan dentro de la empresa. La solución informática fue desarrollada usando
modelos de Proceso de Negocio (BP) en la metodología XP, integrando elementos
de análisis y documentación provistos a través modelos de proceso de negocio

(BPM) usando la notación Business Process Model Notation (BPMN).

151 PLANTEAMIENTO DEL PROBLEMA

SOMEC S.A. es una empresa Vallecaucana dedicada al desarrollo de proyectos y
fabricación de equipos en el sector alimenticio. Dicha empresa está en un
constante crecimiento considerando su amplia gama de servicios y pronto
su nueva línea de producción y expansión a nivel internacional, lo que
provocara mayor flujo de información para el personal.

La organización posee información que es considerada de gran importancia
como datos digitales, físicos, software, hardware, etc., estos se denominan
activos de información, los cuales no tienen un correcto manejo o no están

debidamente gestionados.

La figura 1, muestra el proceso actual de manejo de activos de información en la
empresa, en donde se identificó que los activos en general se encuentran
dispersos en documentos físicos, algunos en World Office y otros están
disponibles en Financi (ambos software contable), estos solo permiten
almacenar la información de activos fijos de la organización, por esto la
información con que se cuenta está dispersa en distintas fuentes, generando
demoras en las consultas de la misma, al no haber un método práctico para el
manejo de los activos de información dentro de la empresa. Algunos de los
activos de información con los que dispone la empresa que no se
encuentran almacenados ni catalogados son: información física y digital,

software, hardware, personas, entre otros.

7.1 SISTEMAS DE INFORMACIÓN

Un sistema de información es un conjunto de elementos que se relacionan entre sí
de acuerdo a unas reglas establecidas, se orienta hacia el tratamiento y la
administración de información, existen varios tipos de sistemas de información
como: sistemas de procesamiento de transacciones, sistemas de información

gerencial, sistema de soporte a decisiones, sistema de información ejecutiva [11].

7.1.1 Tipos de sistemas de información

e Sistema de información gerencial: este sistema resulta de la interacción
entre los diferentes involucrados en procesos dentro de una organización
tales como: personas, tecnologías y procedimientos; estos sistemas son

orientados a dar solución a problemas empresariales.

e Sistema de procesamiento de transacciones: su función es recolectar,
almacenar y recuperar la información generada por transacción producida
dentro de una organización, una transacción es considerado como un
evento que genera o actualiza datos que están almacenados.

e Sistema de soporte a la toma de decisiones: es un sistema de apoyo,
más no que automatiza la toma de decisiones, una decisión es la elección
entre varias opciones basadas en estimaciones de los valores de las
mimas, un sistema de soporte a la toma de decisiones pretende ayudar a
un grupo de personas con un objetivo común a reunir inteligencia, generar
alternativas o caminos y de esta manera tomar decisiones.

e Sistema de información ejecutiva: es una herramienta que está orientada
para usuarios de nivel gerencias, permite monitorear el estado de las
variables de un área de una empresa, partiendo de la información interna y

externa de la misma. [23]

317.2 INGENIERÍA DE REQUISITOS

Con el fin de precisar y llegar a un entendimiento más preciso de lo que es y
representa la ingeniería de requisitos, a continuación se definirá que es un

requisito. Existen varias definiciones, entre las que se pueden citar:

e Una condición o capacidad requerida por un usuario para resolver un
problema.

e Una condición o capacidad que debe alcanzar o poseer un sistema o
componente para satisfacer un contrato, un estándar, una especificación o un
documento formalmente impuesto.

e Una forma documentada de 1 o 2 [24]

Las actividades en Ingeniería de Requisitos (IR) son altamente dependientes de
las decisiones humanas y esto hace difícil automatizarlas. Sin embargo los
analistas de software están siendo cada vez más conscientes de los beneficios
que se pueden obtener del soporte automatizado en la Ingeniería de Requisitos
[25].

Al comienzo de un proyecto de software, la información debe capturarse
rápidamente y de manera comprensiva. Es vital que el ingeniero pueda captar la
mayor cantidad de información ofrecida por el representante de los clientes que
sea posible en periodos cortos de tiempo [26]. Esta actividad se conoce como
especificación de requisitos. Existen especificaciones de requisitos que varían en
estructura, contenido y en formatos de representación. Una de las razones para
esto es la existencia de muchos estándares y guías que definen el contenido y la
estructura de una “buena” especificación de requisitos en diferentes formas. Otra
razón es la influencia del método o métodos usados en el proceso de Ingeniería
de Requisitos [27].

32Otra alternativa para capturar requisitos rápidamente y con más claridad es XP,
que es una metodología ágil centrada en potenciar las relaciones interpersonales
como clave para el éxito en desarrollo de software, promoviendo el trabajo en
equipo. En cuanto a la fase de requisitos en XP se trabajan las historias de
usuario. Se trata de tarjetas de papel en las cuales el cliente describe brevemente
las características que debe poseer el sistema, sean requisitos funcionales o no
funcionales [28]. Por otra parte, Jeffries, Anderson, y Hendrickson, (2001) indican
que se debe pagar un precio por perder la oportunidad de un cliente con alta
disponibilidad para la recolección de información. Respecto a la información
contenida en la historia de usuario, existen varias plantillas sugeridas pero no
existe un consenso al respecto. En muchos casos sólo se propone utilizar un
nombre y una descripción [29] o sólo una descripción, y posiblemente una
estimación de esfuerzo en días [30]. Jeffries, Anderson, y Hendrickson, (2001)
dicen que dependiendo de la complejidad del sistema, debe haber al menos una
historia por cada característica importante, y propone realizar una o dos historias

por programador al mes.

7.3 METODOLOGÍA ÁGIL XP (eXtreme Programming)

Las metodologías ágiles nacen con el objetivo de permitir a los equipos desarrollar
software rápidamente, además responder a los cambios que se presentan a lo
largo del proyecto. Las metodologías agiles se basan en el manifiesto ágil
teniendo siempre como características la inclusión de los clientes en el proceso de
desarrollo. Permite centrarse más en los procesos de desarrollo por medio de
cronogramas flexibles, sin dar mucha relevancia a la documentación. Estas
metodologías son impuestas por el equipo de desarrollo, es decir son amoldadas a
la organización con menos control. El cliente se define como un miembro más del
equipo de desarrollo, los cuales son pequeños, no superiores a 10 integrantes,

trabajando todos en el mismo sitio. Existen pocos roles y no se enfatiza tanto en la

33arquitectura del software [31]. Las metodologías ágiles están centradas en
potenciar las relaciones interpersonales como punto clave para el éxito en el
desarrollo de software, estas promueven el trabajo en equipo y en la fase de
requisitos trabajan con historias de usuario [32]. Lo cual implica que la
comunicación constante entre usuario y desarrollador es vital para describir

claramente las necesidades y objetivos del proyecto [30].

Las metodologías agiles buscan el beneficio común tanto del equipo de desarrollo
como del cliente. El cliente es un miembro más del equipo en estas metodologías,
ya que es quien sabe cómo funcionan los procesos y qué es lo que necesita. La
documentación en las metodologías agiles es más dinámica, está basada en la
comunicación cara a cara con el cliente, evitando intermediarios. Estas
metodologías ágiles buscan aumentar la productividad al generar entregables al
cliente en menor tiempo. El avance del proyecto se mide con mayor confiabilidad
en el número de requisitos aprobados por el stakeholder e implementados. En

este tipo de metodologías, se habla de actividades en lugar de etapas [7].

La metodología ágil XP propone un ciclo de vida dinámico soportado en ciclos de
desarrollo cortos (llamados iteraciones), con entregables funcionales al finalizar
cada ciclo [33]. En cada iteración se realiza un ciclo completo de análisis, diseño,

desarrollo y pruebas (ver figura 5).

XP incluye como estrategia de elicitación de requisitos las historias de usuario, las
cuales describen una funcionalidad que es de valor para el usuario o cliente de un

e Conversaciones acerca de la historia que sirven para profundizar en los

e Pruebas que especifican detalles del documento y que se puede utilizar

desarrollo de software liviana, de bajo riesgo y flexible. Se destaca ante las demás
por ser concreta, ofreciendo retroalimentación en ciclos cortos. Es de enfoque
incremental, permite cronogramas flexibles respondiendo a las necesidades
cambiantes del negocio. Maneja una comunicación directa con el cliente y los

stakeholders. Está diseñada para trabajar con proyectos que serán desarrollados
35por equipos pequeños de programadores donde se efectúan desarrollos que
agregan valor y que pueden ser probados incluso en un día [35].

En [36] indican que el uso de modelos BP, como herramienta para la elicitación de
requisitos en lugar de las historias de usuario, permitió mejorar la comunicación
entre el analista y los stakeholders. Al aumentar el número de requisitos
aprobados por los stakeholders en la primera fase de la metodología XP con
modelos BP, se reduce el tiempo de validación en la fase de planificación, lo que
genera menor tiempo en el desarrollo y entrega de las funcionalidades que son
programadas en el Release Plan. Esta reducción del tiempo en fases posteriores a
la elicitación, se convierte en una oportunidad que mejora la productividad del
desarrollo y que se reduzca el tiempo que toma cada iteración logrando así que se
satisfagan las necesidades de los stakeholders. En este sentido, los modelos
visuales con el uso de los modelos BP permitieron a los usuarios saber cómo
deben usar el sistema y además ayudaron a los stakeholders a entender lo que el
desarrollador propone, manteniéndolos interesados e involucrados en el proceso.
Teniendo en cuenta lo anterior en este trabajo se abordó la metodología XP con el
uso de modelos de proceso en lugar de historias de usuario, basados en que los
modelos de proceso de negocio son una manera más natural de obtener otros
modelos, considerando el proceso como el componente central del sistema de

información a desarrollar.

7.3.1 Fases de la Metodología XP

e Exploración: en esta fase, el cliente define lo que necesita mediante la
redacción de “historias de usuarios” a alto nivel. Los programadores
estiman los tiempos de desarrollo de cada actividad definida en la historia,
con base en esta información [37]. En esta fase las estimaciones
realizadas deben ser preliminares, dado que la especificación es de muy
alto nivel y podrían variar cuando se analicen más en detalle para cada

iteración.

36e Planificación: En esta fase los stakeholder y el grupo de desarrolladores
acuerdan el orden en que deberán implementarse las historias de usuario,
y, asociadas a estas, las entregas. Estas entregas se especifican en un
documento denominado Plan de Entregas, o Release Plan. Además en
esta fase, cuando el desarrollador no tiene claro el requisito, realiza
pequeños programas de prueba denominados spikes, para validar con el
cliente la funcionalidad con el fin de reducir riesgos y facilitar la estimación.
Estos spikes son desechados en la fase de iteración, por lo que se dedica
tiempo a la implementación de los entregables.

e lteraciones: es la fase donde cada historia de usuario definida, se debe
traducir en tareas específicas de programación de funcionalidades, que
generan los entregables. Debido a que las historias de usuario no se
definen con suficiente detalle, en la fase de exploración, entonces se realiza
un proceso de análisis con el cliente para especificar elementos como:
restricciones o datos necesarios para las funcionalidades requeridas.

e Puesta en Producción: implementación de las funcionalidades y
realización de pruebas que puedan generar modificaciones o ajustes,

denominados en esta metodología como fine tuning.

En la metodología XP se utiliza la práctica Release Plan, la cual es un proceso
que crea un plan de alto nivel que cubre un periodo de más de una iteración de
desarrollo. Este plan permite al cliente y al equipo de desarrollo decidir qué será
desarrollado y cuánto tiempo tardará para tener una versión demostrable del

producto. El Release Plan agrupa un conjunto de modelos BP que dan valor a una

parte del negocio y que deberán ser analizadas cada una por tareas [38].

377.4 GESTIÓN DE PROCESOS DE NEGOCIO

Business Process Management (BPM) es un conjunto de métodos, herramientas y
tecnologías utilizados para diseñar, representar, analizar y controlar procesos de
negocio operacionales. BPM es un enfoque centrado en los procesos para mejorar
el rendimiento que combina las tecnologías de la información con metodologías de
proceso y gobierno. BPM es una colaboración entre personas de negocio y
tecnólogos para fomentar procesos de negocio efectivos, ágiles y transparentes.
BPM abarca personas, sistemas, funciones, negocios, clientes, proveedores y

socios [39].

Las personas generalmente usan modelos para respaldar sus conversaciones,
ayudando a la comunicación y comprensión, actuando como respaldo para
prácticamente todos los programas de mejora. Tales modelos forman la base de
una referencia global del negocio, detallando cómo la operación se integra.
Forman parte del material de capacitación y actúan como base para compartir las
buenas prácticas dentro de la Empresa [40].
El identificar los procesos es una tarea primordial y previa a cualquier proyecto
encaminado hacia la mejora de procesos o de BPM. Esta tarea se encamina a
conocer los procesos sobre los que se quiere trabajar, por lo que se debe hacer
una idea sobre que trata de resolver el proceso, qué pretende alcanzar y por qué
existe el proceso, así como el grado de importancia que este tiene para la
organización y para sus clientes. Los principales puntos a identificar en los
procesos son:

e Entradas y Salidas

e Misión del proceso

e Objetivos del proceso

e Clientes del proceso

e Propietario del proceso

e Promotor del proceso
38e Actividades incluidas en el proceso
e Recurso necesarios del proceso
e Alcance y límites del proceso

e Indicadores para la medida del proceso

Proceso de Negocio
Un conjunto de una o más actividades relacionadas, que colectivamente logran un

objetivo de negocio, normalmente dentro del contexto de una estructura

organizacional definiendo roles y relaciones [41].

7.4.1 Ciclo de Vida BPM

En la figura 6 se muestra las actividades del ciclo de vida BPM. Normalmente se
parte de un análisis de la situación actual de los procesos empresariales
(Monitorización de los Procesos Actuales, recogiendo algunos indicadores de
referencia) que indicará lo que se desea mejorar para conseguir unos resultados
empresariales. Una vez se conoce que se tiene que desarrollar un proyecto BPM,
se comienza a Modelizar y Diseñar Procesos de Negocio, creando lo que se
denomina como Arquitectura Empresarial (se detecta el mapa de procesos de la
empresa y se modelizan los procesos para su automatización, así como se
definen los nuevos indicadores a controlar para orientarse hacia los objetivos de
negocio). En la Automatización e Integración, se ejecutan los procesos de negocio
utilizando motores de Workflow y soluciones de integración de aplicaciones y de
datos. Según se van ejecutando los procesos de negocio, se va controlando el
comportamiento mediante la monitorización (detectando cargas de trabajo, cuellos
de botella, ineficiencias, buenos resultados, puntos de mejora...). En la
monitorización se detectan mejoras a realizar, por lo que se empieza de nuevo el

ciclo revisando la modelización y haciendo los ajustes necesarios de diseño [4-2].

7.4.2 BPMN

En BPMN (Business Process Model and Notation), los “Procesos de Negocio”
involucran la captura de una secuencia ordenada de las actividades e información
de apoyo. Modelar un Proceso de Negocio implica representar cómo una empresa

realiza sus objetivos centrales [40].

BPMN es un lenguaje formal que permite modelar, simular y eventualmente,
ejecutar procesos de negocios. Su sintaxis está basada en elementos gráficos,
pero tales elementos tienen una relación uno a uno con instrucciones del lenguaje
BPEL, lo cual permitirá generar código ejecutable BPEL a partir de un modelo en
la notación BPMN así como modificar este código haciendo uso del modelado
gráfico en BPMN. El estándar BPMN fue publicado en el año 2004 por el Business
Process Management Initiative (BPMI) y posteriormente adoptado por el OMG
como especificación en el año 2006, habiendo sido adoptado como estándar por

todo tipo de organizaciones y fabricantes de software.

417.5.1 ORM (Mapeo Objeto Relacional)

El mapeo objeto-relacional es una técnica de programación para convertir datos
del sistema de tipos utilizado en un lenguaje de programación orientado a objetos
al utilizado en una base de datos relacional. En la práctica esto crea una base de
datos virtual orientada a objetos sobre la base de datos relacional. Esto posibilita
el uso de las características propias de la orientación a objetos (esencialmente la

herencia y el polimorfismo)[44].

7.5.2 DAO (Data Access Object)

El DAO implementa el mecanismo de acceso requerido para trabajar con la base
de datos. Se utiliza para abstraer y la encapsular todos los accesos a la base de
datos. El DAO gestiona la conexión con la fuente de datos para obtener y
almacenar datos. Esencialmente, la DAO actúa como un adaptador entre el

componente y la base de datos|45].

7.5.3 JPA y JPQL

La API de persistencia de Java (JPA) es una solución basada en estándares Java
para la persistencia. La persistencia utiliza un enfoque de mapeo objeto /
relacional para cerrar la brecha entre un orientado a objetos modelo y una base de
datos relacional. La API de persistencia de Java también se puede utilizar en Java
SE aplicaciones, fuera del entorno Java EE. Java Persistence consiste en lo
siguiente áreas:

e La APl Java Persistence
e Ellenguaje de consulta (JPQL)
e Objeto / metadatos de mapeo relacional

427.5.4 EJB

Los Entreprise Java Beans son componentes del lado del servidor que encapsulan

la lógica del negocio de una aplicación[44].

7.5.5 Managed Bean

Son objetos gestionados por contenedor de peso ligero (POJO) con requisitos
mínimos, para apoyar a un pequeño conjunto de servicios básicos, como la
inyección de recursos, las devoluciones de llamada de ciclo de vida, y los
interceptores. Beans gestionados representan una generalización de los beans
gestionados especificados por la tecnología JavaServer Faces y se puede utilizar

en cualquier lugar de una aplicación Java EE, no sólo en la Módulos Web [46].

7.5.6 JSF (Java Server Faces)

La tecnología Java Server Faces establece el estándar para la construcción de
interfaces de usuario del lado del servidor. Con las aportes de un grupo de
expertos, la API Java Server Faces fue diseñados para que puedan ser
aprovechados por las herramientas que harán que el desarrollo de aplicaciones

web sea incluso más fácil| 47].

7.5.7 PrimeFaces

Es una librería de componentes visuales de código abierto para el conjunto Java
Server Faces 2.0 desarrollada y mantenida por Prime Technology. Su objetivo
principal es ofrecer un conjunto de componentes para facilitar la creación y diseño

de aplicaciones web[48].

7.5.8 Servidor de Aplicaciones Glassfish

GlassFish es un servidor de aplicaciones desarrollado por Sun Microsystems que
implementa las tecnologías definidas en la plataforma Java EE y permite ejecutar

aplicaciones que siguen esta especificación[49].

8 DESARROLLO DEL PROYECTO

XP es una de las llamadas metodologías ágiles de desarrollo de software más
exitosas de los tiempos recientes. La metodología propuesta en XP está diseñada
para entregar el software que los clientes necesitan en el momento en que lo
necesitan. XP alienta a los desarrolladores a responder a los requerimientos
cambiantes de los clientes, aún en fases tardías del ciclo de vida del desarrollo
[33]. Por lo anterior, se hizo uso de esta metodología, la cual se detalla a
continuación y en la cual se agregó el componente de gestión de procesos de
negocio en lugar de las historias de usuario.

Los modelos de proceso de negocio, aunque agregan un componente formal a la
documentación de los requisitos, se proponen como estrategia ya que es una
notación fácil de entender, no solo para los analistas sino para los involucrados. A
pesar que puede generar mayor tiempo en la primera iteración de la metodología
este se verá reducido en fases como la de planificación o ejecución en donde no
se tiene que validar con el cliente y realizar recodificación [36]. En [36] indican
que, el uso de modelos de proceso de negocio como apoyo en el proceso de
elicitación de requisitos, permite aumentar la cantidad de requisitos aprobados por
los involucrados en un 28% y el uso de los modelos de proceso de negocio en
lugar de historias de usuario, genera una diferencia de 30% de requisitos

aprobados.

Gestión de Incidentes

El proceso habitual de gestión de incidentes es el siguiente:

- Detección y registro del incidente: Con la afectación a uno oO varios
usuarios, O la detección de un sistema de monitoreo, se crea una nueva
incidencia.

- Clasificación y soporte inicial: Como pueden recibirse múltiples incidencias
al mismo tiempo, el paso siguiente es determinar el nivel de prioridad, para
enviarse al personal de soporte correspondiente. La prioridad se asigna
según: Impacto: Afectación del negocio y/o número de usuarios afectados,

Urgencia: Tiempo máximo para solución y/o nivel de servicio.

En primer lugar, desde la capa del cliente y capa web, se hizo uso de JSF que es
un estándar de Java hacia la construcción de interfaces de usuario para
aplicaciones web que simplifican el desarrollo de aplicaciones web del lado del
cliente. En las páginas JSF se utilizó la extensión *.xhtml, es decir, una
combinación de XML con HTML, en donde se pudo incluir componentes como
CSS, JavaScript, entre otros.

La especificación de JSF define seis fases distintas en su ciclo de vida:

1. Restauración de la vista: Crea un árbol de componentes en el servidor para

representar la información de un cliente.
572. Aplicar valores de la petición: Actualiza los valores del servidor con datos del
cliente.

3. Proceso de validación: Valida los datos del usuario y hace la conversión.

4. Actualización de valores del modelo: Actualiza el modelo del servidor con
nuevos datos.

5. Invocar la aplicación: Ejecutar cualquier lógica de aplicación para cumplir con la
solicitud.

6. Procesar la respuesta: Guarda un estado y da una respuesta al cliente.

Para facilitar el uso de los componentes se hizo uso de PrimeFaces que es una
librería de componentes visuales de código abierto para el conjunto Java Server
Faces 2.0 desarrollada y mantenida por Prime Technology. Su objetivo principal es
ofrecer un conjunto de componentes para facilitar la creación y diseño de
aplicaciones web [51]. Los componentes de PrimeFaces fueron conectamos con el
servidor a través de los ManagedBean que no son más que beans que reciben la

información del html para enviarlos a la capa de lógica.

En segundo lugar, para la capa de servicio, que en este trabajo se llamó Capa de
Lógica, se hizo uso de los JavaBeans empresariales (Enterprise JavaBeans [1, 6],
EJB), que son una tecnología (API) que forma parte del estándar de Java EE.
Están diseñados para desarrollo y despliegue de aplicaciones (distribuidas) de
negocio basadas en componentes del lado del servidor. Una vez que se desarrolla
una aplicación, ésta puede ser desplegada en cualquier servidor que soporte la

especificación de EJB [48].

En tercer lugar, para la capa de persistencia se hizo uso de JPA que es, una
abstracción que está por encima de JDBC lo que permitió ser independiente de
SQL. Todas las clases y anotaciones de esta API se encuentran en el paquete

javax.persistence.

58Los principales componentes de JPA son:

* Mapeo de base de datos relacionales (ORM). Es el mecanismo para mapear
objetos a los datos almacenados en una base de datos relacional.

* Un API administrador de entidad para realizar operaciones en la base de datos
tales como crear, leer, actualizar, eliminar (CRUD).

e El Java Persistence Query Language (JPQL) que permite recuperar datos con un
lenguaje de consultas orientado a objetos.

e Las transacciones y mecanismos de bloqueo cuando se accedan a los datos

concurrentemente, la API Java Transaction (JTA).

El motor de base de datos usado fue MYSQL.

Detección y corrección de errores

Cuando se encuentra un error (“bug”), éste fue ser corregido inmediatamente, y se
tienen precauciones para que errores similares no vuelvan a ocurrir. Asimismo, se
generaron nuevas pruebas para verificar que el error haya sido resuelto. Este tipo
de pruebas no fueron documentadas, dado que fueron realizadas por el
desarrollador en el momento de la codificación de tareas de cada iteración.

Pruebas de aceptación

Finalmente, se realizaron las pruebas de aceptación que fueron creadas en base a
los modelos BP, en cada ciclo de la iteración del desarrollo. En este caso, el

66cliente especificó uno o diversos escenarios para comprobar que un modelo BP
fue correctamente implementado. Estas pruebas fueron consideradas, “pruebas de
caja negra” ya que los clientes fueron los responsables de verificar que los
resultados de éstas pruebas sean correctos. Un modelo BP no se pudo considerar
terminado hasta tanto pasará correctamente todas las pruebas de aceptación.

en el proyecto, a medida que transcurría el tiempo los requisitos tuvieron cambios
pero gracias a la metodología se pudo adecuar de buena manera el proyecto,

visual a través de una notación, respaldando las conversaciones entre el equipo
de desarrollo y los stakeholders, con el propósito de aumentar la productividad del

desarrollo y la calidad de los proyectos de software a desarrollar.

Evaluados los procesos llevados a cabo para el desarrollo de SISTEMA DE
INFORMACIÓN WEB PARA LA GESTIÓN DE ACTIVOS DE INFORMACIÓN A
